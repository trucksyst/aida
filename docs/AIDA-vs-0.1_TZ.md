# Техническое задание: AIDA-vs 0.1

## 1. Введение

### 1.1. Название проекта
AIDA Virtual Server (AIDA-vs) 0.1

### 1.2. Цель
Основная цель проекта — реорганизация архитектуры существующего браузерного расширения AIDA в модель "клиент-сервер". Это позволит отделить пользовательский интерфейс (UI) от основной логики сбора и обработки данных (бэкенд), превратив бэкенд в "виртуальный сервер".

Такой подход решает следующие задачи:
- **Гибкость:** UI может быть любым веб-приложением, а не только страницей расширения.
- **Масштабируемость:** Бэкенд-сервер может работать независимо от браузера, на более мощной машине.
- **Централизация:** Вся логика и управление данными находятся в одном месте (на сервере), а не в браузерах разных пользователей.
- **Стабильность:** Процессы сбора данных (харвестеры) не будут зависеть от открытой вкладки в браузере.

### 1.3. Краткое описание архитектуры
Система будет состоять из двух основных, слабо связанных компонентов:
1.  **Сервер (Backend):** Приложение на Node.js, которое выполняет всю основную работу: запускает харвестеры для сбора данных, управляет аутентификацией, хранит данные и предоставляет API для взаимодействия с клиентом.
2.  **Клиент (Frontend):** Веб-приложение (SPA - Single Page Application), которое является пользовательским интерфейсом. Оно взаимодействует с сервером через HTTP API для получения данных и отправки команд.

## 2. Архитектура системы

### 2.1. Сервер (Backend) - "Виртуальный сервер"

Это ядро системы.

- **Технологический стек:**
  - **Среда выполнения:** Node.js
  - **Веб-фреймворк:** Express.js (для создания API)
  - **Веб-скрейпинг:** Playwright или Puppeteer (для запуска харвестеров и эмуляции действий в браузере на стороне сервера)

- **Модули ("Блоки"):**
  1.  **API-сервер:** Основной шлюз для всех входящих запросов от клиента. Отвечает за маршрутизацию, обработку запросов и аутентификацию.
  2.  **Модуль аутентификации:** Управляет входом в систему, сессиями и cookies. Обеспечивает защиту эндпоинтов API.
  3.  **Основной модуль (Core Logic):** Аналог `background.js`. Управляет состоянием системы, очередями задач для харвестеров и координирует работу других модулей.
  4.  **Модули-харвестеры:** Адаптированные версии скриптов из папки `/harvesters`. Вместо работы в качестве content-скриптов в браузере пользователя, они будут запускаться сервером в экземплярах Playwright/Puppeteer для доступа к сайтам-источникам.
  5.  **Модуль хранения (Storage):** Управляет хранением данных (собранная информация, настройки, кэш). Может быть реализован на основе файлов (JSON) или легковесной базы данных (например, SQLite).

### 2.2. Клиент (Frontend) - "UI-блок"

Это пользовательский интерфейс, полностью отделенный от логики.

- **Технологический стек:** HTML, CSS, JavaScript (ванильный, без фреймворков, для преемственности с `ui/app.js`).
- **Принцип работы:**
  - Является одностраничным веб-приложением (SPA).
  - Вся динамика и взаимодействие с пользователем обрабатываются на клиенте.
  - Все действия, требующие данных или выполнения операций (например, запуск сбора), инициируют асинхронные HTTP-запросы (через `fetch`) к API сервера.
  - `ui/app.js` будет полностью переписан для работы с API вместо `chrome.runtime.sendMessage` и других API расширений.

## 3. API (Application Programming Interface)

Сервер должен предоставлять следующий минимальный набор эндпоинтов.

- **Формат обмена данными:** JSON
- **Аутентификация:** Все запросы, кроме `/login`, должны быть защищены и требовать активной сессии (проверка по cookie).

### Эндпоинты:

- `POST /api/login`
  - **Назначение:** Аутентификация пользователя на целевых сервисах (DAT, TruckerPath и т.д.) через серверный браузер.
  - **Тело запроса:** `{ "service": "dat", "credentials": { "user": "...", "pass": "..." } }`
  - **Ответ:** `{ "success": true, "message": "Logged in successfully" }`. Устанавливает http-only cookie сессии.

- `GET /api/status`
  - **Назначение:** Получение текущего статуса системы и харвестеров.
  - **Ответ:** `{ "harvesters": { "dat": "idle" | "running" | "error", "truckstop": "logged_out" } }`

- `POST /api/harvest/start`
  - **Назначение:** Запуск процесса сбора данных.
  - **Тело запроса:** `{ "source": "dat" | "truckstop" | "truckerpath" }`
  - **Ответ:** `{ "success": true, "message": "Harvesting started for [source]" }`

- `GET /api/data`
  - **Назначение:** Получение собранных данных.
  - **Параметры запроса (query):** `?source=dat&limit=50`
  - **Ответ:** `{ "data": [ ...массив объектов с данными... ] }`

## 4. Требования к реализации

1.  **Создать структуру проекта для сервера:**
    - `package.json` с зависимостями: `express`, `playwright`, `cookie-parser`, `body-parser`.
    - Папки: `/src/api`, `/src/harvesters`, `/src/core`, `/storage`.
2.  **Адаптировать харвестеры:** Переписать логику из `harvesters/*.js` для работы в среде Node.js с использованием Playwright. Логика скрейпинга остается прежней, меняется только способ взаимодействия со страницей.
3.  **Реализовать API-сервер:** Настроить Express, создать все эндпоинты и логику их обработки.
4.  **Реализовать UI-клиент:**
    - UI-ресурсы (html/css/js) могут отдаваться как статика самим Express-сервером.
    - Переписать `ui/app.js`: удалить весь код, связанный с `chrome.*` API, и заменить его на вызовы `fetch` к эндпоинтам `/api/*`.
5.  **Настроить запуск:** Создать скрипты для запуска сервера (`npm start`) и сборки проекта.
